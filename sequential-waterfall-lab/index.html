Sequential Waterfall Lab

This page issues a strictly sequential chain of small fetches so the waterfall should appear as clean stair-steps. It is meant for isolating browser-side delay between requests, especially when you suspect CPU affinity, event-loop starvation, or resource balancing side effects.

Use Run Network-Only Chain first. Then use Run CPU-Between-Requests Chain to amplify any browser scheduling weakness without changing the server behavior.

Ready.
How It Works 50 sequential requests to one tiny payload URL with cache-busting query strings and cache: 'no-store'.
Modes
  • Network-only: fires the next request immediately after the previous one finishes.
  • CPU-between-requests: adds main-thread CPU work between requests.
Read The Results If request durations stay similar but the gap before the next request grows, the delay is likely browser-side rather than network-side.
URL Controls
  • ?mode=network or ?mode=cpu
  • &requests=50 to &requests=200
  • mode=cpu uses cpudelay=75 by default
  • &cpudelay=25, &cpudelay=75, or &cpudelay=150 to override CPU mode
  • cpudelay is ignored in mode=network
  • Examples: ?mode=cpu, ?mode=cpu&cpudelay=25, ?mode=cpu&requests=100&cpudelay=75, ?mode=cpu&requests=200&cpudelay=150
